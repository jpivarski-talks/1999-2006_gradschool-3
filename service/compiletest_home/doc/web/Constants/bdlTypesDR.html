   UInt32 CellID

   float  pedestal_RMS
}

    Explanation:
    Charge pedestals and widths as measured in a DR BigCal. One for each wire; units are in TDC counts. The "charge pedestal" is the digitized response of the charge circuit when there is no input signal.

  • DRadcsaturation
    Type: Constants
Name: DRadcsaturation
{ 
   UInt32 CellID
   float  saturation
   float  saturation_RMS
}

    Explanation:
    Charge signal in TDC counts when the charge circuit is saturated. One for each wire. Measured in a DR Big Cal by injecting a huge amount of charge and recording the charge circuit's response. This is the largest possible charge signal that can be recorded, and is used to distinguish time hits from charge hits. So-called "seed thresholds" (see DRFBUSChannel) are computed from these measured values.

  • DRWeightLayer
    Type: Constants
Name: DRWeightLayer
{
   UInt32 layer
   float  fittingweight
}

    Explanation:
    This is the inverse average resolution squared for each layer. The absolute fitting weight.
    Note that the other "DRWeight*" entities are multiplicative corrections to this number with various binnings.

    J. Pivarski: the value of "fittingweight" above has always been a multiplicative correction on the standard DR resolution on 123 microns (e.g. if fittingweight for a hit is 2, the resolution of that hit will be taken to be 246 microns (doubled, not quadrupled), and the hit will be weighted less in the fit).

    Fit chi2 = Sum (residual_i)2 / ("fittingweight"_i * 123 microns)2

    DRWeightLayer, DRWeightDrift, DRWeightTheta and DRWeightPhi are valid for CLEO-III runs (104100 - 132285, which is data3 - data29), and are used by code from releases July10_03 (pre-ZD) and earlier. CLEO-c data must use DRWeight2LayerDriftEntAng, DRWeight2LayerTheta and DRWeight2LayerPhi.

  • DRWeightDrift
    Type: Constants
Name: DRWeightDrift
{
   UInt32 indexNumber
   UInt32 layer
   UInt32 bin
   float  fittingweight
}

    Explanation:
    Renormalization of fitting weight dependent on drift distance. We will include details of the data format (binsize, interpolation method) later.

    J. Pivarski: binsize is determined by "fake layers" 0 and 48: 

    1 48    1    minValue
2  0 numbins maxValue
    where minValue and maxValue define the applicable range in unsigned drift distance, normalized to cell width (1.0 is the cell boundary for a zero-entrance angle track), and numbins sets the number of bin divisions.

    The interpolation method gives bin values in the middles of the bins, a linear interpolation between them, and a linear extrapolation outside of the range. (Yes, it's also an extrapolation in the low half of the first bin and the high half of the last bin.)

    DRWeightLayer, DRWeightDrift, DRWeightTheta and DRWeightPhi are valid for CLEO-III runs (104100 - 132285, which is data3 - data29), and are used by code from releases July10_03 (pre-ZD) and earlier. CLEO-c data must use DRWeight2LayerDriftEntAng, DRWeight2LayerTheta and DRWeight2LayerPhi.

    Type: Constants

    Explanation:

  • DRWeight2LayerDriftEntAng
    Type: Constants
Name: DRWeight2LayerDriftEntAng
{
   UInt32 layer
   UInt32 driftbin
   UInt32 entangbin
   float resolution
}

    Explanation:
    This is the main constants type for assigning hit resolutions for CLEO-c onward. These hit resolutions are assigned as a function of three variables: DR layer number, unsigned drift distance (normalized to the cell size at zero entrance angle), and the sine of the entrance angle. Since the last two parameters are continuous, the hit resolution is determined via a 2-D linear interpolation, where the bin values themselves are given at the centers of the bins. Outside of the defined range, resolutions are given as an extrapolation (so be sure the last two drift bins have non-negative slope!).

    Validity ranges of drift and entang, as well as numbers of bin divisions, are set using a "fake layer zero", like this:

    layer    driftBin  EntAngBin  Resolution
  0       1          0           0.0 (min absDrift value)
  0      15 (max)    0           1.5 (max absDrift value)
  0       0          1          -1.0 (min sinalpha value)
  0       0         20 (max)     1.0 (max sinalpha value)

    "resolution" is a multiplicative factor on 123 microns: a track fit chi2 is defined like this:

    Fit chi2 = Sum (residual_i)2 / ("resolution"_i * 123 microns)2

    Note that DRWeight2LayerTheta and DRWeight2LayerPhi can also affect hit resolution.

  • DRWeight2LayerTheta
    Type: Constants
Name: DRWeight2LayerTheta
{
   UInt32 layer
   UInt32 thetabin
   float resolution
}

    Explanation:
    Since the resolution of a hit also depends on dip angle, this constants type provides a multiplicative correction. Bins of cos(theta) are interpolated/extrapolated in the same way as DRWeight2LayerDriftEntAng. By convention, these corrections should always be normalized such that cos(theta) == 0 has a "resolution" value of 1.0. This way, the layer-by-layer dependence is characterized only by DRWeight2LayerDriftEntAng.

    Validity range and number of bins in cos(theta) are set using a "fake layer zero", like this:

    layer    thetaBin  Resolution
  0       1          -1.0 (min cosTheta value)
  0      20 (max)     1.0 (max cosTheta value)

    This correction doesn't need to be set by hand (from data). If DRWeight2LayerTheta contains only 

    layer    thetaBin  Resolution
  0       0           0.0
    then a "standard" correction of (1 - cos(theta)2)-1/4 will be applied. This is proportional to the square root of the track length through a drift cell.

  • DRWeight2LayerPhi
    Type: Constants
Name: DRWeight2LayerPhi
{
   UInt32 layer
   UInt32 phibin
   float resolution
}

    Explanation:
    The only justification for this constant is that it provides for future flexibility in assigning hit resolutions. Bins of phi are interpolated cyclically, and the resolution correction (usually 1.0, I hope) is multiplied to that given by DRWeight2LayerDriftEntAng and DRWeight2LayerTheta.

    The number of bins is set using a "fake layer zero", like this:

    layer    phiBin    Resolution
  0      20 (max)      0.0

    This correction doesn't need to be set by hand (from data). If DRWeight2LayerPhi contains only 

    layer    thetaBin  Resolution
  0       0           0.0
    then no correction will be applied (which is what you probably want to do!).

DROnline

  • DRFBUS1887S
    Type: Constants
Name: DRFBUS1877S
{
   UInt16  CSR0
   UInt32  CSR1
   UInt16  CSR18
   UInt16  BroadcastClass
}

    Explanation:
    The contents of the control and status registers (CSRs) of the LeCroy Fastbus 1877S TDCs in the DR Fastbus crates. These settings configure the boards for normal operation, controlling things like sparsification, number of hits stored per event, start/stop mode, and so on.

  • DRFBUSChannel
    Type: Constants
Name: DRFBUSChannel
{
   UInt16  ChannelAddress  
   UInt16  DaqId
   UInt16  Crate           SelectBy
   UInt16  SparsThreshold
   UInt16  SeedThreshold
}

    Explanation:
    The information used by Fritz and the CPU in a DR Fastbus crate to process each raw TDC hit before putting it in the event stream. The channel address is a bit-packed code that contains the Fastbus crate, slot, and channel. Thus this word and the DaqId determine one part of the DR electronics map, from Fastbus channel to DaqId. (See DRElecMap for more.) This map is used to label each raw hit with a unique address describing from where it came. (For handling miscablings, see DRDaqCellIDMap.) The sparsification threshold is used in the TDC board itself to discard charge hits below a given threshold, typically just above pedestal (see DRadcpedestal). The seed threshold is used by Fritz to flag those hits that are above a particular level. In the DR, hits above the seed threshold are guaranteed to be time hits, since the threshold is set at the maximum possible charge value (see DRadcsaturation).
    See the ConstantsMapping code for the details of the bit-packing.

  • DRTQTBoard
    Type: Constants
Name: DRTQTBoard
{
   UInt16  BoardAddress
   UInt16  Crate           SelectBy
   UInt8   CalEnable
   UInt8   CalDriverEnable
   UInt16  Preamp0Voltage
   UInt16  Preamp1Voltage
   UInt16  Preamp2Voltage
   UInt16  Preamp3Voltage
   UInt16  TriggerPulseWidth
   UInt8   TriggerMode
   UInt32  KillRegisterHi
   UInt32  KillRegisterLo
   UInt32  TestRegisterHi
   UInt32  TestRegisterLo
}

    Explanation:
    Constants used by DRSbus to configure a TQT board for data-taking or a BigCal. The BoardAddress is a bit-packed word combining the TQT crate and slot number. The other quantities are as follows:
    CalEnable Determine what combination of the four preamps attached to a board are to be pulsed in a calibration cycle
    CalDriverEnable A single bit that enables the circuit that produces the calibration pulse that is sent to the preamps during calibration pulsing (a BigCal)
    PreampNVoltage An 8-bit setting controlling the power supplied to each of the four preamps attached to a single TQT board
    TriggerPulseWidth An 8-bit setting that controls how much the trigger timing pulse is stretched before being sent to the tracking trigger.
    TriggerMode Controls the operation of the trigger FPGA on the TQT board
    KillRegister A 64-bit register inside the trigger FPGA used for testing or to mask bad (noisy) channels
    TestRegister A 64-bit register inside the trigger FPGA used for testing ot to force on dead channels

  • DRTQTChannel
    Type: Constants
Name: DRTQTChannel
{
   UInt16  ChannelAddress
   UInt16  Crate              SelectBy
   UInt16  TimeThreshold
   UInt16  TriggerThreshold
}

    Explanation:
    Constants used by DRSbus to configure each TQT channel. The ChannelAddress is a bit-packed word that combines the TQT crate, slot, and channel into a single word. The thresholds are 8-bit values that control the discriminator threshold on the timing circuits for a channel. A separate threshold is applied for the trigger to reduce noise without reducing hit efficiency.

  • DRTSTrigBoard
    Type: Constants
Name: DRTSTrigBoard
{
   UInt16   index
   String   Component    SelectBy
   String   BoardName
   UInt16   L1width
   UInt16   L1delay
   boolean  L1enable
   UInt16   CALwidth
   UInt16   CALdelay

}

    Explanation:
    Constants used by the DR Timing System to set up its Trig board. The InternalTrigEnable bit controls whether a special stand-alone mode is used, where software commands can be used to trigger the system. In normal operation (data-taking or BigCal), the trigger system provides the triggers, "internal triggering" is disabled, and none of these settings are used.   • DRTSGateBoard
    Type: Constants
Name: DRTSGateBoard
{
   UInt16   index
   String   Component    SelectBy
   String   BoardName
   UInt16   FBUSCrate
   UInt16   G1width
   UInt16   G1delay
   UInt16   G2width
   UInt16   G2delay
   UInt16   G3width
   UInt16   G3delay
   UInt16   G1enable
   UInt16   G2enable
   UInt16   G3enable
   UInt16   CALenable
   UInt16   InvertMask
}

    Explanation:
    Constants used by the DR Timing System that determine the width and delay of various gating signals sent to the Fastbus and TQT crates to coordinate digitization of drift chamber data. For more information, see the Timing System documentation. Note that the widths and delays are the raw values as stored in the registers, which happen to determine not the width and delay of each gate, but (0x3FF - width or delay) of each gate. Sorry, that's the way it is. So widths will become smaller linearly as the width register value is increased, etc.
    The FBUSCrate field is not used.
    The InvertMask (a 6-bit register) allows one to determine the polarity of the signals sent out by the timing system. (Positive or negative logic, basically.)
    The various Enables (each a 6-bit register) determine which of the four possible signals (G1-3, CAL) are output on each of the 6 connectors on a given Gate board.

DRMiscellaneous

  • DRElecMap
    Type: Constants
Name: DRElecMap
{
   UInt16   DaqId
   UInt16   TQTChannelAddress
   UInt16   PreAmpChannelAddress
}

    Explanation:
    Contains the rest of the information required to build a complete electronics map for the drift chamber. DRFBUSChannel above contains the DaqId to Fastbus mapping, and this contains the mapping to TQT channel and preamp channel. Each of the "address" words above is again a bit-packed word combining crate, slot, and channel (TQT) or layer, sector, channel-in-sector (preamp) information into a single 16-bit word.
    See the ConstantsMapping code for the details of the bit-packing.
